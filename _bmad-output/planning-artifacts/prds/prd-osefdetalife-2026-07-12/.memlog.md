updated: 2026-07-13T19:57

- (decision) Vision: Simple booking system for small restaurants that want to focus on cooking. White-label, embeddable widget, flat-fee, no per-cover commission.
- (decision) Feature 1 (Booking Widget): FR-1 through FR-11. Widget landing (embed, name/address, "Book now"), party size selection (1-8), date selection (open dates only), time selection (15-min slots), booking details form (name, email, custom field), booking submission (Firestore, confirmation), back navigation, loading states, error handling (invalid slug, Firebase down), responsive design (fixed width), demo page (standalone HTML).
- (decision) Feature 2 (Restaurant Dashboard): FR-12 through FR-27. Dashboard home (view today's bookings, navigate dates, empty state, real-time updates), Settings — Restaurant Info (edit name, configure hours, edit address, timezone), Settings — Table Groups (add, edit, delete), Settings — White Label (primary color, secondary color, custom field), Settings — Account (change password, sign out).
- (decision) Feature 3 (Restaurant Onboarding): FR-28 through FR-43. Email/password sign up, Google sign up, login page, guided setup flow, restaurant name, slug, address, opening hours, table groups, white-label colors, custom field, skip optional steps, onboarding completion, embed code page, copy embed code, demo page link.
- (decision) Feature 4 (Firebase Infrastructure): FR-44 through FR-53. Restaurant profile data model, table groups data model, opening hours data model, bookings data model, slug resolution, security rules (public read, owner write, diner booking), compute-on-read availability, slug uniqueness enforcement.
- (decision) Non-Goals: No integrations, payments, AI, multi-location, advanced booking, communication, analytics, mobile apps, internationalization, table management.
- (decision) Success Metrics: SM-1 (widget load < 2s), SM-2 (booking completion > 80%), SM-3 (onboarding < 10min), SM-4 (5 restaurants in first month), SM-5 (50 bookings/restaurant/month).
- (decision) Open Questions: All resolved with recommendations. Cancellation policy (direct contact), max party size (table capacity), lead time (30 days), slot granularity (15min), advance window (30 days), confirmation (auto-confirm), slug format (lowercase + hyphens), logo (not in MVP).
- (decision) PRD finalized. Status set to final. 53 FRs across 4 features. Ready for Epics & Stories.
- (fix) Context reset after Feature 2 caused FRs 1-27 to be regenerated incorrectly. User provided correct FRs 1-11 (Booking Widget) and 12-27 (Restaurant Dashboard). Features 3-4 confirmed correct. PRD and HTML updated with correct FRs.
- (decision) Party mode challenge: 3 issues resolved. (1) Widget auth model: FR-51 changed to unauthenticated booking creation with Firestore shape validation. Diner cannot read or edit bookings for MVP. OTP-based editing deferred to v2. (2) Timeline: extended from 20h to 30h to keep all FRs. (3) Data model: FR-46 hours changed from subcollection to Record on restaurant document. FR-47 bookings removed phone and dietary fields. Added assumptions: hours are weekly recurring (no holiday overrides), diner creates without auth.
