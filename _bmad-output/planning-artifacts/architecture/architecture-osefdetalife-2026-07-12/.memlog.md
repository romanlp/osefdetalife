- (decision) AD-1: Widget deployment via script tag. Binds: widget deployment. Prevents: npm/iframe approaches. Rule: widget renders via custom element on any website including WordPress.
- (decision) AD-2: Direct Firebase (Firestore) from browser. Binds: all data access. Prevents: API layer overhead. Rule: widget and dashboard use Firebase client SDK directly, security rules enforce access control.
- (decision) AD-3: Firebase Auth with email/password + Google Sign-In. Binds: restaurant owner authentication. Prevents: passwordless/other providers. Rule: dashboard auth uses Firebase Auth, supports email/password and Google OAuth.
- (decision) AD-4: Web Components with Shadow DOM for widget. Binds: widget rendering. Prevents: style conflicts with host page. Rule: widget renders as custom element with Shadow DOM isolation.
- (decision) AD-5: Compute on read for availability. Binds: booking availability logic. Prevents: pre-computed slot storage. Rule: availability calculated by querying bookings for date, subtracting from table groups. 15-minute slots.
- (decision) AD-6: No table splitting. Binds: availability calculation. Prevents: multi-table party accommodation. Rule: party of N requires single table of capacity >= N.
- (decision) AD-7: Auto-confirm bookings for MVP. Binds: booking status model. Prevents: pending/confirmation flow. Rule: bookings confirmed on submission, status set to confirmed. Confirmation step deferred to next iteration.
- (decision) AD-8: Diner collects name + email + optional custom field. Binds: booking form. Prevents: anonymous bookings. Rule: diner name and email required, custom field optional with restaurant-defined label.
- (decision) AD-9: Restaurant slug for identification. Binds: widget deployment. Prevents: UUID-based identification. Rule: each restaurant has unique slug used in widget embed (booking-widget restaurant='slug').
- (decision) AD-10: Widget flow: party size → available times → date selection → details form → confirmation. Binds: widget UX. Prevents: date-first flow. Rule: diner selects party size first, then sees available times for that party size.
- (decision) AD-11: Dashboard shows today's bookings with date picker. Binds: dashboard UX. Prevents: all-bookings view. Rule: default to today, date picker to navigate to other dates.
- (decision) AD-12: Booking actions in model, not in UI for MVP. Binds: booking status model. Prevents: premature UI complexity. Rule: status field supports confirmed/cancelled, UI read-only for MVP, actions added in next iteration.
- (decision) AD-13: One restaurant per account for MVP. Binds: account model. Prevents: multi-location complexity. Rule: single restaurant per auth account, multi-location deferred.
